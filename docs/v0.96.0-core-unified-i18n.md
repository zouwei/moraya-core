# v0.96.0 — Unified i18n at `@moraya/core/i18n`

**Version**: v0.96.0
**Status**: 🚧 Phase 1 delivered (engine + tests + placeholders) — Phase 2-5 pending
**Platform**: `core` (shared markdown editor + i18n npm package)
**Owner repo**: `moraya-core/` (this repo) — consumed by `moraya/` (PC desktop) and `moraya-web/` (Mobile + Web)

---

## 1. Background

PC desktop (`moraya/`) and Mobile+Web (`moraya-web/`) each ship their own i18n engine and locale JSON. This session — and prior ones — surfaced three concrete consequences:

1. **Recurring missing-key bugs.** The AI settings page on Web rendered raw `settings.ai.usage.title` strings because Web's JSON misnested those keys under `settings.ai.local.*`. Fixed in-session by hoisting; no CI gate caught it.
2. **Schema drift.** PC has 43 top-level namespaces, Web has 46. Only 12 overlap. Even within the overlap, key names differ — PC uses `editor.sourceMode` (camelCase), Web's overlapping namespace uses snake_case variants. No common contract exists.
3. **Language coverage gap.** PC ships 12 locales (en, zh-CN, zh-Hant, ar, de, es, fr, hi, ja, ko, pt, ru). Web ships 2 (en, zh-CN). Switching PC→Mobile in Spanish silently downgrades to English.

The fix is to consolidate the engine and the locale data into one source of truth. The user (confirmed via in-session AskUserQuestion) chose:

- **Location**: embed in `@moraya/core` as a subpath export `@moraya/core/i18n` (do not yet create a separate `@moraya/i18n` package).
- **Scope**: full migration — engine + merged JSON + canonical key naming applied across both consumers in one iteration.
- **AI helpers**: include `resolveForLocale` / `resolveAllLocales` (originally PC-only) in the shared engine.

Architecture constraint: keep `src/i18n/` strictly isolated inside core so a future move to a standalone `@moraya/i18n` package is a path rename, not a refactor.

## 2. Design decisions

| Decision | Choice | Reason |
|---|---|---|
| Package boundary | Subpath export `@moraya/core/i18n` | Avoids spawning a new npm package now; preserves single-source-of-truth; future extraction is mechanical (path rename in consumers + `mv src/i18n moraya-i18n/src`). |
| Framework coupling | Zero | Engine uses a custom `Readable<T>` primitive (`store.ts`) whose `subscribe` signature is Svelte-compatible — consumers can pass it into `derived()` from `svelte/store` directly, but the package itself imports no svelte. Preserves future extraction. |
| `t()` shape | Plain function `t(key, vars?) → string` | Matches Web's existing API exactly; PC's templates wrap into a Svelte `derived` store inside the consumer-side shim to preserve `$t('foo')` reactivity. |
| Locale loading | Lazy per-language via `import('./locales/<loc>.json')` switch in `loader.ts` with module-scoped cache | Each consumer's bundler (Vite, etc.) can code-split per language so users only download the active locale. (Phase 1 currently inlines into the tsup output — see "Open: per-locale code-splitting" below.) |
| Persistence | Pluggable `PersistenceAdapter` callback at `initLocale()` | PC wires Tauri plugin-store; Web wires localStorage; engine stays host-agnostic. |
| Canonical naming | `snake_case` for terminal segments, `.` for hierarchy | Matches Web's mobile-era keys (`mobile.appbar.untitled`, `compose.drafts.new_sub`). Smaller blast radius migrating PC's older `editor.sourceMode` / `aiBudget.*` than the reverse. |
| Test test-doubles | `__setLoader` injection on `loader.ts` | Lets unit tests install fixture loaders without real JSON files; resets the cache as a side effect. Marked `@internal` so the public type surface excludes it. |
| English fallback | Always loaded once active locale ≠ en | `t()` resolves: active → English → literal key. Missing translations stay visible in dev. |
| `__mt: true` sentinel | Per-namespace machine-translation flag | Marks blocks that were machine-translated as initial baseline; surface for follow-up human review without bloating per-key metadata. |

## 5. Codemod tool (Phase 3-4 prerequisite)

`moraya-core/scripts/i18n-rename.mjs` — NEW.

- Input: `mapping.json` produced from Phase 2 merge.
- AST-aware: `@babel/parser` for TS, `svelte/compiler` for templates.
- Rewrites string literals inside `t('…')`, `t("…")`, `` t(`…`) ``, `resolveForLocale('…', …)`, `resolveAllLocales('…', …)`.
- Skips comments and MDX content.
- Outputs: per-file diff summary + unresolved cases list.
- Idempotent — re-running is a no-op if no new mappings.

## 6. CI gates (Phase 5)

### `scripts/i18n-coverage.mjs`

- Walks consumer source tree, extracts the set of `t('…')` key arguments.
- Walks merged JSON, extracts the set of defined dotted keys (via `flattenMessages`).
- **Fails** when any referenced key is undefined.
- **Warns** (non-fatal) when any defined key has zero references — guards against bit-rot, but no auto-prune (some keys are referenced dynamically).
- Wired into GitHub Actions for both moraya and moraya-web.

### `scripts/i18n-isolation.mjs`

- Greps `import` statements inside `src/i18n/**` of moraya-core.
- **Fails** when any import resolves to a path outside `src/i18n/**`.
- **Fails** when any import names `prosemirror-*`, `markdown-it`, `svelte`, or `react`.
- Wired into moraya-core's GitHub Actions on PR.

## 8. Open / follow-ups

### Per-locale code-splitting

Currently `tsup` with `splitting: false` inlines all `import('./locales/<loc>.json')` targets into `dist/i18n/index.js`. With placeholder JSONs the bundle is ~12KB; once Phase 2 fills the real content (~80KB × 12 locales), index.js will be ~1MB.

The consumer's Vite bundler may still code-split — TBD when Phase 3 lands. Fallback options:

1. tsup `loader: { '.json': 'copy' }` — emit JSONs as separate files in `dist/i18n/locales/` and rewrite loader to fetch via runtime path.
2. Switch to `splitting: true` for the i18n entry only — requires a separate tsup config (tsup doesn't support per-entry splitting flags directly).
3. Use a build-time esbuild plugin that intercepts `./locales/*.json` paths and emits them as chunks.

To evaluate after Phase 3.

### `t` reactivity in PC

PC's existing 1,716 callsites use `$t('foo')` (Svelte derived-store auto-subscribe). The shared engine ships a plain function. Consumer-side shim adds the Svelte wrapper:

```ts
// In moraya/src/lib/i18n/i18n.ts post-migration:
import { t as tFn, locale as coreLocale } from '@moraya/core/i18n'
import { derived } from 'svelte/store'
export const t = derived(coreLocale, () => tFn)
// Other exports re-exported as-is
export { setLocale, initLocale, ... } from '@moraya/core/i18n'
```

This preserves the `$t('foo')` template pattern with zero codemod work on the reactivity dimension.

### Web reactivity question

Web's current `t` is a plain function — components don't re-render on locale switch. Acceptable today (locale changes are rare and usually accompanied by a full reload). Phase 3 could optionally wrap with `derived` too, OR keep as-is. Decision deferred.
