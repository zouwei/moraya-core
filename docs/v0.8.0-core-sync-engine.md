# v0.8.0 — 共享文档同步引擎 `@moraya/core/sync`

## 背景

Moraya Web 在线编辑 markdown 时,连续输入频繁误报「检测到冲突」(实际没有其他设备在编辑)。根因不是缺少版本基准,而是各端各自实现的 autosave 循环里的竞态:

1. **保存无在途互斥** —— base etag 只在往返成功后推进,两次重叠保存携带同一过期 base,服务器判 `REMOTE_NEWER`,即**本设备自冲突**。
2. **熔断过敏** —— 自动和解内部又触发保存,重叠持续,和解连续两次即放弃、弹手动对话框。
3. **etag 退化** —— 服务器响应缺 `updatedAt` 时 etag 退化,下次省略 base,必然 `BASE_MISSING`。

同时,PC 早已有一套成熟的 KB↔Picora 同步引擎(v1.6.0,298 测试:三方 diff 决策表 + node-diff3 行级 3-way merge + 富冲突面板),但只在桌面端。本迭代把同步与合并策略下沉到 `@moraya/core`,PC / Web / Mobile 三端统一,并借机把 Web 的 autosave 重写为一个正确的状态机。

## 交付

新增子路径导出 `@moraya/core/sync`,两层皆平台无关:

### 1. 纯合并原语(自 PC 逐字下沉)

- `threeWayDiff(last, local, remote, maxFileSizeBytes, initialAuthority='local')` —— 目录级决策表(upload / download / delete-local / delete-remote / conflict / aligned / skip-large;首次同步无 base 时按 `initialAuthority` 裁决而非粗暴冲突)。
- `threeWayMergeLines(local, base, remote)` / `twoWayMergeLines(local, remote)` / `assembleMerged(result, picks, defaultPick)` / `conflictChunkCount(result)` —— git 式行级合并。3-way 用 `node-diff3` 的 `diff3Merge`(`excludeFalseConflicts`:两侧相同改动不算冲突);无 base 回退用 `diff` 的 `diffLines`。返回统一的 `{ chunks, hasConflict, mergedText }`,冲突块携带 `local`/`remote`/`base` 行数组供逐块 pick。
- 类型:`ManifestEntry` / `LocalManifestEntry` / `Local|RemoteManifest` / `DiffAction` / `DiffResult` / `MergeChunk` / `MergeResult` / `ChunkPick` / `ConflictEntry` / `ConflictResolution` / `InitialAuthority`(命名保持与 PC 一致,PC 侧改为 re-export shim 后 import 路径零改动)。

### 2. `DocSyncEngine` —— 单文档乐观并发状态机

修复 Web 误报的核心。模型 = **乐观并发(条件写)+ 客户端 single-flight 串行化 + git 式 3-way 自动合并**,单用户多设备场景的行业标准做法,无需 OT/CRDT。

**三条不变式**杜绝误报:

- **single-flight**:任一时刻至多一个写在途。写在途时 `flushNow()` 加入同一 promise,绝不发起第二个携带过期 base 的重叠写。
- **base 原子推进**:base 是 `{ etag, content }` 对,仅在写往返成功后同轮推进,etag 与 content 永不漂移。
- **trailing-write 循环**:写在途期间缓冲又变,循环立即以新 base 再写一次,不丢尾键、不靠防抖竞态。

真·服务器分歧时的 `reconcile()` 管线:① 内容相同只是基线过期 → 采纳 etag 收敛;② 本地自 base 未改 → 远端胜(`onAutoMerged`);③ 双方都改 + 有 base + clean 3-way → 自动合并后以远端为新基线重提;④ 真同处冲突 / 无 base → `onConflict` 交 UI。**熔断改为「轮内无进展」判据**(同轮内连续两次 conflict 且服务器 etag 未动才升级人工),取代旧的跨 flush `mergeStreak` 计数 —— single-flight 后自冲突消失,熔断只对真外部并发写触发,且每轮 flush 重置;另设 `MAX_RECONCILE_ROUNDS = 4` 防活锁。

**IO 依赖注入**(参照 AITransport):core 只出同步逻辑,消费端实现 `DocSyncIO` 执行读写。约定 **IO 不抛异常**,以判别联合返回(`ok` / `missing` / `offline` / `error` / `conflict`)—— 错误分类是平台职责,core 保持纯净,也避免跨包 `instanceof`。

```ts
interface DocSyncIO {
  read(): Promise<{ type: 'ok'; content: string; etag: string } | { type: 'missing' } | { type: 'offline' } | { type: 'error'; error: unknown }>
  write(content: string, baseEtag: string | null): Promise<{ type: 'ok'; etag: string } | { type: 'conflict' } | { type: 'offline' } | { type: 'error'; error: unknown }>
}

class DocSyncEngine {
  seed(content, etag)            // 载入既有文档为初始 base
  queue(content)                // 防抖入口(默认 1500ms)
  flushNow(): Promise<void>     // Cmd+S:清 timer + 立即;在途则加入
  resolveConflict(content, remoteEtag): Promise<void>
  dispose()
  // 回调:onStatusChange / onSaved / onAutoMerged / onConflict / onOffline / onError
  // 选项:debounceMs / shouldCreate(空文档创建守卫)
}
```
